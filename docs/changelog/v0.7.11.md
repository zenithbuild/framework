# zenith-docs v0.7.11

Framework hardening, release-integrity, editor-DX, and maintainability release for the Zenith lockstep package train.

## Highlights

- The compiler now fails earlier and more consistently for invalid script syntax, invalid markup expressions, clearly unbound markup identifiers, malformed markup, and direct-call event handler forms.
- Remote image optimization validates and pins resolved remote targets across requests and redirects.
- Runtime cleanup is more deterministic around structural fragment replacement, throwing disposers, failed effects, and embedded-markup URL validation.
- Dev server stability improved across rebuilds, failed builds, resource routes, static-to-interactive transitions, route add/delete flows, and port conflicts.
- Release, package, CI, and dependency checks now have stronger fail-closed coverage before publish.
- Public docs and generated types now align more closely with the server route contract, advisory client route policy naming, signed cookie guarantees, and config truth.

## Compiler And Diagnostics

- Event handlers reject direct-call forms more consistently while preserving valid identifier/member references and inline functions.
- Script and expression syntax errors now use stable compiler diagnostics: `ZEN-SCRIPT-SYNTAX` and `ZEN-EXPR-SYNTAX`.
- Clearly unknown markup roots now fail with `ZEN-EXPR-UNBOUND`.
- Malformed markup now reports `ZEN-MARKUP-PARSE` diagnostics instead of panic-style bridge output.

## Runtime And Server Hardening

- Structural fragment replacement drains nested cleanup before replacing DOM regions.
- Cleanup continues after throwing disposers and reports collected errors deterministically.
- `zeneffect` setup and cleanup registration failures roll back active subscriptions.
- Embedded markup URL-bearing attributes, including `srcset`, are decoded and normalized before protocol validation.
- SSE metadata validation now rejects unsafe `event`, `id`, and `retry` values while preserving valid multiline data framing.
- Packaged Node servers can use `ZENITH_PUBLIC_ORIGIN` or `publicOrigin` so secure cookies are set correctly behind HTTPS termination.

## Build, Routing, And Dev Server

- Route classification is shared so protected `guard` / `load` / `action` routes cannot be marked prerender in one build path and missed in another.
- Dev output honors `basePath`, including SPA navigation under the configured prefix.
- Dev rebuild cache invalidation now tracks route deletion and interactivity-shape changes more reliably.
- Server route output names are collision-resistant.
- Browser refresh after failed or stale dev builds gets clear build-error responses without wedging the dev server.
- Dev port conflicts fall forward to the next available port and report the requested, occupied, and final serving ports without killing processes.

## Package, Release, And CI Integrity

- Windows CLI bin entrypoint detection and package `.bin` ownership are covered.
- Native binary publish smoke checks fail closed.
- CI integration shard coverage is derived from the filesystem, and the router legacy Jest suite is part of the core gate.
- Release workflow trust boundaries were audited and hardened with least-privilege permissions, scoped OIDC, fail-closed release checks, and static workflow policy tests.
- Dependency audit cleanup was limited to dev/test/browser automation tooling and same-major transitive patch-line updates.

## Docs, Types, And Config Truth

- Docs gates catch forbidden component callsite DOM event prop drift while preserving ordinary callback props.
- Client route policy APIs are documented as advisory UX only; server `guard(ctx)` and `load(ctx)` remain the security boundary.
- Router event declarations now match emitted route events.
- Generated `Zenith.LoadContext` declarations come from the real server route contract and do not expose unsupported stream/SSE fields.
- Auth docs now state that signed session cookies provide integrity and tamper resistance, not confidentiality.
- Public config accepts documented `static-export`, keeps core/CLI config parity locked, rejects stale `devTrace` config input, and documents that dev config changes require restarting `zenith dev`.
- CLI README wording now matches hosted image endpoint support for `node`, `vercel`, `netlify`, and `static-export`.

## Middleware, Plugin Surface, And Maintainability

- Route-local `withMiddleware(...)` parity is covered end-to-end across page routes, resource routes, dev, packaged Node, and hosted function paths without introducing global middleware.
- Archived legacy plugin code is quarantined from the current public API surface, and public docs state that the plugin API is currently closed.
- The oversized-file refactor plan now records current audit state, split boundaries, risk notes, validation commands, and focused follow-up issues.

## Editor Tooling

- The framework source tree now carries the aligned language-server/editor DX work, while npm release ownership for `@zenithbuild/language` and `@zenithbuild/language-server` remains in their standalone repositories.
- LSP smoke coverage verifies initialization, compiler diagnostics, hover, and completion over stdio, with Neovim smoke coverage and docs that state current limitations.
- Standalone `zenith-language` and `zenith-language-server` repositories have separate `v0.7.12` tags. Their npm publish state remains separate from this framework release and was blocked locally by npm auth.

## Published Versions

- Lockstep train `0.7.11` covers `@zenithbuild/bundler`, `@zenithbuild/cli`, `@zenithbuild/compiler`, `@zenithbuild/core`, `@zenithbuild/router`, and `@zenithbuild/runtime`.
- Optional platform packages for `@zenithbuild/bundler-*` and `@zenithbuild/compiler-*` move with the same `0.7.11` train.
