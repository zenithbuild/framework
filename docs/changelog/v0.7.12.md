# zenith-docs v0.7.12

Component Server Values v1 release for the Zenith lockstep package train.

## Highlights

- Component Server Values let layouts and components own local server values without turning components into route handlers.
- The internal Scoped Server Data pipeline now covers architecture/metadata locks, owner discovery, diagnostics, manifest classification, server-output packaging, request-time execution, SSR template binding, hydration, and generated type declarations.
- Scoped data works across Node, dev, preview, Vercel, and Netlify server paths while preserving route-owned `guard(ctx)`, `action(ctx)`, and `load(ctx)`.
- Repeated component instances receive deterministic scoped runtime keys, static literal props are supported for scoped component data, and dynamic scoped props fail clearly.
- Final regression fixtures lock SSR, hydration, hosted parity, static client leak exclusion, diagnostics, serialization, and generated declarations.
- Config-time V1 plugins can apply narrow validated config patches, while transform, middleware, compiler, bundler, route/security, and adapter plugin hooks remain unsupported.
- Release automation now has stronger framework publish-surface alignment and native platform smoke/recovery checks before tagging and publishing.

## Component Server Values

- Layout and component owners can declare top-level server constants in `<script server lang="ts">` and use those values in the same owner template.
- Advanced owners can export `data(ctx, props)` for owner-local scoped data that depends on request context and static literal props.
- Runtime keys identify layout owners, singleton component owners, and repeated component instances without merging owner-local data into page route data.
- Component Server Values are not React Server Components and do not add component/layout route-control APIs.

## Scoped Runtime, SSR, And Hydration

- Scoped owner modules are emitted under server output and excluded from static client bundles.
- Request-time execution runs after successful route guard/load resolution and serializes scoped slices into the SSR payload.
- The bundled SSR template path resolves layout and component values from owner-local scoped slices while preserving legacy flat route payload compatibility.
- Hydration consumes serialized scoped payloads only; it does not refetch, refresh, or re-execute server values on the client.
- Scoped owner failures and non-serializable scoped returns remain fatal instead of silently omitting data.

## Type Declarations And Diagnostics

- Builds now emit `.zenith/zenith-scoped-server-data.d.ts` alongside existing Zenith declaration files.
- Generated declarations expose owner-key and runtime-key maps with conservative inference and safe fallbacks.
- Diagnostics cover invalid owner APIs, reserved Level 1 bindings, prerender conflicts, unsupported scoped component props, and static-target conflicts.
- Final fixture coverage keeps `has_load` route-owned and `has_scoped_server_data` scoped-owner-specific.

## Docs, Migration, And Groundwork

- The Component Server Values guide explains the public feature name, the internal Scoped Server Data contract, Level 1 and Level 2 syntax, SSR/hydration behavior, generated declarations, and migration boundaries.
- Server-data docs now distinguish route-owned request control from owner-local component/layout values.
- Public AI docs artifacts were regenerated for the updated Component Server Values, server data, routing, props, hydration, and diagnostics docs.
- Adapter and middleware changes in this train are groundwork/stabilization only: this release does not ship middleware arrays, nested scopes, controlled headers, route-check participation, or a public adapter plugin API.
- The adapter plugin surface RFC is planning documentation only; it does not make adapter plugins a public API in this release.
- create-zenith starter projects now ignore generated Zenith output.

## Config And Release Integrity

- `zenith.config.*` now accepts named config-time V1 plugins with an optional `config()` hook.
- Plugin config patches are validated, shallow, deterministic, and limited to safe config keys.
- Unsupported plugin hooks such as `transform`, `middleware`, `server`, `compiler`, `bundler`, route/security policy hooks, and adapter hooks are rejected.
- Framework publish-surface checks remain scoped to the lockstep framework packages and intentionally exclude standalone-owned editor packages.
- Native platform binary smoke checks and publish recovery checks were tightened for the framework release workflow, including Linux bundler container smoke coverage.

## What This Release Does Not Change

- No React Server Components-style server/client component split.
- No client scoped refetch, cache/revalidation API, background refresh, or build-time scoped prerender.
- No page-level server variables.
- No component/layout `guard`, `load`, `action`, `redirect`, or `deny`.
- No public adapter plugin API and no middleware B1-B4 expansion.

## Published Versions

- Lockstep train `0.7.12` covers `@zenithbuild/bundler`, `@zenithbuild/cli`, `@zenithbuild/compiler`, `@zenithbuild/core`, `@zenithbuild/router`, and `@zenithbuild/runtime`.
- Optional platform packages for `@zenithbuild/bundler-*` and `@zenithbuild/compiler-*` move with the same `0.7.12` train.
