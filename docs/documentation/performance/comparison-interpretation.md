# Comparison Interpretation

This document defines how Zenith benchmark results should be interpreted once public measurements are published.

## What Is Being Measured

This page does not define a single benchmark. It defines how published benchmark outputs should be read across bundle, build, startup, rebuild, and runtime measurements.

## What The Metric Represents

Interpretation guidance explains what a result can support, what it cannot support, and where caution is required before generalizing it.

## Why It Matters

Benchmarks lose trust when they are used to imply a universal ranking that the methodology never actually proved.

## Zenith Benefits If Proven

If Zenith shows strength in a measured area, that can support its fit for workloads that value those characteristics. It should not be translated into “best at everything.”

## Tradeoffs and Constraints

Frameworks optimize for different constraints. Some results will favor compiler-first design, while others may favor mature ecosystems, broader abstractions, or different runtime choices.

## Interpretation Notes

Published comparisons should always state where Zenith appears well-suited, where tradeoffs remain, and where another framework may still be the better fit.
